## 1. 🎯 本节内容

- 分仓库模式

## 2. 🫧 评价

这篇笔记记录了在设计 TNotes 的时候，为何决定要分仓库来管理笔记，以及仓库划分的策略。

## 3. 🤔 为什么要分仓库来管理笔记？直接将所有的学习笔记合并到一个仓库中不行吗？

### 3.1. 核心问题

若所有笔记都统一丢到一个仓库中，后续仓库体积和笔记数量必然会变得异常庞大，不便于管理。

在单仓库模式下：

- 全量的 git push、git clone、git pull 操作耗时会显著增加
- github actions 跑自动构建也会变得极其缓慢，每次 push 完之后需要等很长时间才能在 github pages 中完成部署
- 随着体积和数量变多，TNotes 执行常用命令 `tn:update`、`tn:dev` 耗时必然变长，极其影响书写体验
- ……

在分仓库模式下：

- 基于分仓库的模式，可以确保任意单一知识库都能独立运行，独立部署
- 当使用一个临时设备的时候，也可以更方便的单独 clone 指定的仓库，拉下来之后安装下依赖，改完后直接 push 就完事儿了，不需要 care 其它知识库的状态，TNotes.xxx 彼此之间是互相独立的

真实需求场景：比如现在正在学习算法和数据结构，此时通常只需要关注和算法数据结构直接相关的笔记即可，如果所有笔记混在一个 git 仓库中，就无法很好地去定位相关笔记。

### 3.2. 当前的笔记架构

```markmap

- TNotes（根知识库）
  - TNotes.algorithms（子知识库）
  - TNotes.c（子知识库）
  - TNotes.canvas（子知识库）
  - ……

```

每一个节点都是一个 git 仓库，所有仓库的笔记目录汇总到根知识库 TNotes 中。

## 4. 🤔 知识库的划分标准是？

因人而异，这个问题没有统一的答案，比如 `TNotes.canvas`、`TNotes.electron`、`TNotes.typescript` 技术栈都可以直接划分到 `TNotes.front-end` 前端这个知识库中，也可以拆分为独立的知识库存在。

这里记录一下各人的处理策略，在创建知识库的时候可以考虑以下问题，若满足，则定义为一个独立的知识库。

问题：是否有对应的书籍或系统的视频教程？

示例：`TNotes.typescript` 就问问自己市面上是否有对应的书籍或者视频教程来专门介绍 typescript 呢？

如果答案是肯定的，那么就直接创建一个独立的知识库来管理和 typescript 相关的笔记。

如果答案是否定的，或者很少的资料来专门介绍它，那么说明它很可能是一个比较散的知识点，大概率是属于某个更大的知识体系下的子级，不应该将其定义为一个独立的知识库。
